# 交易日判断机制说明

守护进程如何判断"今天是否美股交易日"以及"操作哪个交易日"。

## 核心问题

系统在**北京时间**触发(12:00 决策 / 22:00 执行),但美股的交易日历、开盘时间、SEC 财报都按**美东时间**计。必须正确把"北京时刻"映射到"美股交易日(session_date)",且**决策和执行必须落到同一个 session_date**,否则执行阶段会找不到决策生成的目标文件。

## 两层判断

### 第一层:`resolve_session_date(now_cn)` — 操作哪个交易日

把北京时刻映射到美股 session 日期:

```
北京 12:00-23:59  → session_date = 北京日期当天
北京 00:00-11:59  → session_date = 美东日期(回退一天)
```

**为什么 12:00-23:59 用北京日期?**

实测验证:在北京 12:00 和 22:00,无论夏令时还是冬令时,要交易的美股 session 都落在与北京日期**相同的日历日**。用北京日期保证了:
- ✅ 决策(12:00)和执行(22:00)的 session_date **永远一致** → 文件名匹配,链路不断
- ✅ 夏令时、冬令时都正确

| 北京时间 | session_date | 美股实际 session |
|---------|-------------|-----------------|
| 夏令时 12:00 决策 | 06-27 | 06-27 ✓ |
| 夏令时 22:00 执行 | 06-27 | 06-27 ✓ |
| 冬令时 12:00 决策 | 12-15 | 12-15 ✓ |
| 冬令时 22:00 执行 | 12-15 | 12-15 ✓ |

**为什么 00:00-11:59 回退到美东日期?**

北京凌晨(如手动 `--run-once` 在 03:00)时,北京日期比美东超前一天。此时回退到美东日期,锚定到"最近收盘的那个 session",避免日期超前。

### 第二层:`_session_is_tradable(session_date)` — 是否交易日

拿到 session_date 后,判断它是否美股开市,由 `--trading-day-source` 控制:

| 模式 | 判断方式 | 识别节假日 |
|------|---------|-----------|
| **`alpaca_calendar`** (默认) | 查 Alpaca `/v2/calendar` API | ✅ 是 |
| `weekday` | 仅判断周一~周五 | ❌ 否 |
| `always` | 永远 True | ❌ 仅测试 |

**默认 `alpaca_calendar`** 最可靠:
- 查询 Alpaca 官方交易日历
- 自动识别周末 + 所有美股节假日(感恩节、圣诞、独立日等)
- 结果缓存在内存(`calendar_cache`),同一天不重复查 API

## 完整流程

```
主循环每 60 秒:
  now_cn = 当前北京时间
  session_date = resolve_session_date(now_cn)   # 第一层:哪个交易日

  if not _session_is_tradable(session_date):     # 第二层:是否开市
      skip                                        # 周末/节假日跳过

  if now_cn.time() >= 12:00:  run decision        # 决策
  if now_cn.time() >= 22:00:  run execute         # 执行
```

## 如何查看判断结果

心跳日志每 30 分钟打印一次,同时显示北京时间、美东时间、session_date:

```
[Scheduler] heartbeat 2026-12-15T12:30:00+08:00 (US 2026-12-14 23:30 EST) session=2026-12-15
```

也可在 Dashboard 的 Settings 标签页查看当前 session 状态。

## 边界情况总结

| 场景 | 行为 |
|------|------|
| 正常 12:00 决策 | session = 北京日期,查日历,开市则跑 |
| 正常 22:00 执行 | session = 北京日期(与决策一致),跑 |
| 周末 | Alpaca 日历返回空 → 跳过 |
| 美股节假日 | Alpaca 日历返回空 → 跳过 |
| 凌晨手动补跑 | session 回退到美东日期(最近 session) |
| 显式 `--date YYYY-MM-DD` | 用指定日期,跳过自动解析 |

## 历史修复

**之前的隐患**(已修复):原代码 `session_date = now_cn.date()` 直接用北京日期。在 12:00/22:00 窗口正确,但**凌晨手动 `--run-once` 会比美东日期超前一天**,可能查错交易日。现在 `resolve_session_date()` 在凌晨窗口回退到美东日期,消除该风险,同时保持 12:00/22:00 的决策/执行一致性。
